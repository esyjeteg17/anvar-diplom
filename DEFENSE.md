## Слайд 5. Технологический стек

**Тезисы:**

| Слой       | Технология                             | Обоснование                                  |
|------------|----------------------------------------|----------------------------------------------|
| Frontend   | Nuxt 4 (Vue 3)                         | Современный SSR-фреймворк, экосистема Vue   |
| Язык       | TypeScript (strict)                    | Типобезопасность, ловит ошибки на этапе компиляции |
| Стили      | Tailwind CSS v4                        | Utility-first, быстрая вёрстка, дизайн-система через `@theme` |
| State      | Pinia                                  | Официальный store для Vue 3, типизация из коробки |
| Карты      | Leaflet + OpenStreetMap                | Open-source, без API-ключей                  |
| Backend    | Django 5 + DRF                         | Зрелый фреймворк, готовая админка, миграции |
| База       | PostgreSQL 14+                         | Надёжная реляционная СУБД, поддержка JSON   |
| Auth       | Django session + CsrfExempt            | Простота, безопасность через CORS-allowlist |

**Текст речи:**

> Для frontend я выбрал Nuxt 4 с TypeScript в строгом режиме — это
> современный SSR-фреймворк на базе Vue 3, обеспечивающий хорошее SEO
> для лендинга и удобную разработку SPA для админки. Стилизация —
> Tailwind CSS четвёртой версии, в которой токены дизайн-системы
> объявляются прямо в CSS через директиву `@theme`. Для управления
> состоянием — Pinia, официальный стор для Vue 3. Карта на лендинге
> построена на Leaflet с тайлами OpenStreetMap — это open-source решение,
> работающее без API-ключей. Бэкенд — Django пятой версии с Django REST
> Framework, база данных — PostgreSQL. Аутентификация — на базе сессий
> Django, защита от CSRF реализована через строгий список разрешённых
> origins в CORS, что является приемлемой моделью для внутренней
> административной панели.

## Слайд 6. Архитектура системы

**Что показать:** диаграмма из трёх блоков с двумя стрелками к бэкенду.

```
┌──────────────────┐         ┌──────────────────┐
│  frontend-       │         │  frontend-       │
│  landing         │         │  admin           │
│  (порт 4040)     │         │  (порт 4041)     │
│  Публичный сайт  │         │  Закрытая часть  │
└────────┬─────────┘         └────────┬─────────┘
         │ REST API                   │ REST API
         │ (форма заявки)             │ (CRUD + auth)
         ▼                            ▼
       ┌────────────────────────────────┐
       │       backend (Django)         │
       │       порт 8042                │
       │                                │
       │  /api/leads/                   │
       │  /api/centers/                 │
       │  /api/children/                │
       │  /api/lessons/                 │
       │  /api/finance/                 │
       │  /api/equipment/               │
       │  /api/staff/                   │
       │  /api/auth/                    │
       └─────────────┬──────────────────┘
                     │
                     ▼
            ┌────────────────┐
            │  PostgreSQL    │
            │  anvar_db      │
            └────────────────┘
```

**Текст речи:**

> Система разделена на три независимых компонента: публичный лендинг,
> административная панель и общий бэкенд. Лендинг и админка — это два
> независимых SPA-приложения на Nuxt 4, развёрнутые на разных портах.
> Они общаются с единым REST API на Django через стандартный HTTP +
> JSON. Backend, в свою очередь, хранит данные в PostgreSQL. Такое
> разделение даёт несколько преимуществ: каждую часть можно
> масштабировать и обновлять независимо; публичный лендинг не зависит
> от логики авторизации админки; одна команда сущностей и одна точка
> правды для бизнес-данных.

**Что показать:** ER-диаграмма (минимум: Center, Child, Staff, Lesson, Attendance, Expense).

**Текст речи:**

> «Центр»: к ней привязаны и дети, и занятия. Каждое занятие содержит

## Слайд 8. Публичный лендинг

**Тезисы:**

- 12 секций: Navbar → Hero → лента → О школе → Цифры → Занятия → лента →
  Центры → Преподаватели → Квиз → Контакты → Footer.
- Интерактивная карта четырёх центров на Leaflet с кастомными метками
  в виде брендовой шестерёнки.
- Квиз из 10 вопросов с генерацией промокода; промокод автоматически
  передаётся в форму заявки.
- Анимации появления при скролле через IntersectionObserver.
- Брендовый дизайн: палитра из 5 цветов, шестерёнки и LEGO-кубики
  как мотивы, фирменный шрифт Inter.

**Текст речи:**

> Публичный лендинг состоит из двенадцати секций, идущих сверху вниз.
> Главный экран содержит логотип SBORIKA с шестерёнкой вместо буквы
> «O», заголовок и CTA-кнопку с пульсирующим оранжевым свечением.
> На фоне — вращающиеся шестерёнки и парящие изометрические LEGO-кубики;
> при скролле они смещаются с разной скоростью, создавая эффект
> параллакса.
>
> Секция «Центры» содержит интерактивную карту на Leaflet. Каждая метка
> — это кастомная SVG-шестерёнка бренда. При наведении показывается
> плашка с точным адресом. Реализован один глобальный обработчик через
> делегирование событий — это надёжнее, чем вешать listener на каждую
> метку, и устойчиво к hot module replacement в режиме разработки.
>
> Секция «Квиз» — это интерактивный тест из 10 вопросов о
> предпочтениях семьи. По завершении квиза пользователь получает
> промокод на бесплатное пробное занятие. Реализован отдельный
> Pinia-стор `useLeadIntentStore`, который хранит этот промокод между
> секциями: при клике «Записаться» промокод автоматически подставляется
> в форму контактов и отправляется в бэкенд вместе с заявкой.
>
> Анимации появления реализованы через собственный composable
> `useReveal` на базе IntersectionObserver — без сторонних библиотек.
> Stagger-эффекты — через проп `delay` у компонента-обёртки `<Reveal>`.

## Слайд 9. Админ-панель

**Тезисы:**

- Защищённая часть, доступ через `/login`.
- Глобальный middleware `auth.global.ts` редиректит неавторизованных
  пользователей на страницу входа.
- 4 рабочие вкладки: Финансы, Расписание, Дети, Оборудование.

**Что показать:** скриншоты каждой вкладки.

**Текст речи:**

> Административная панель — это второе SPA-приложение, защищённое
> через сессию Django. На уровне роутинга работает глобальный
> middleware: если пользователь не авторизован — он перенаправляется
> на страницу логина. После входа создаётся sessionid-cookie и все
> последующие запросы идут с заголовком `credentials: include`.
>
> В админке четыре основных раздела:
>
> **Финансы.** Дашборд с четырьмя ключевыми показателями: доход
> (потенциальный, считается из активных абонементов), зарплаты,
> расходы за период и итоговая прибыль. Сводка отдаётся бэкендом одним
> агрегатным запросом через action-эндпоинт `summary`. Ниже — две
> таблицы с CRUD сотрудников и расходов; при изменении любой записи
> сводка автоматически перезагружается.
>
> **Расписание.** Собственный недельный календарь — без сторонних
> библиотек. Семь столбцов-дней с кнопкой «+» для добавления занятия,
> карточки занятий с темой и статусом «проведено». В форме редактирования
> занятия — список детей выбранного центра с чекбоксами «был» и
> комментариями преподавателя. Каждая отметка автосохраняется по
> событию `change` или `blur` через стор. Логика «PUT, если запись
> существует, иначе POST» инкапсулирована в одном методе стора.
>
> **Дети.** Реестр клиентов школы с поиском по ФИО и телефону, фильтром
> по центру, и полным CRUD через модальное окно.
>
> **Оборудование.** Простой счётчик с кнопками `+1` и `−1` для каждой
> категории: наборы, планшеты, подставки. На бэкенде — отдельный
> action-эндпоинт `adjust` с защитой от ухода в отрицательные значения.

## Слайд 10. Архитектурные решения и инженерные практики

**Тезисы:**

- Компонентный подход с максимальной декомпозицией: один универсальный
  компонент с пропсами вместо нескольких похожих.
- Единая дизайн-система: 5 цветовых токенов и шрифт Inter в `@theme`,
  никакого хардкода hex.
- Pinia-сторы по доменной модели; экшены инкапсулируют API-вызовы
  и обработку ошибок.
- Toast-уведомления как сквозной механизм UX-фидбэка для всех
  CRUD-операций.

**Текст речи:**

> Один из ключевых принципов, заложенных в архитектуру — максимальная
> декомпозиция на компоненты с параметризацией через пропсы.
> Например, компонент `IconGear` для шестерёнки имеет 6 пропсов:
> размер, количество зубцов, цвет, цвет центра, радиус отверстия и
> скорость вращения. Это один компонент покрывает все варианты
> шестерёнки на сайте — от логотипа до фоновых декораций. Аналогично
> универсальные `BrandButton`, `IconCard`, `Modal`, `FormField`.
>
> Дизайн-система централизована: все цвета бренда вынесены в CSS-переменные
> `--color-bg`, `--color-brand`, `--color-line`, `--color-slate`,
> `--color-ink` через директиву `@theme` Tailwind 4. Tailwind автоматически
> генерирует utility-классы вида `bg-brand`, `text-ink`, и в коде
> компонентов нигде не встречается прямой hex.
>
> Pinia-сторы организованы по доменам: один стор на сущность. Каждый
> экшен внутри стора инкапсулирует HTTP-запрос, обработку ошибок DRF,
> локальное обновление состояния и вывод toast-уведомления. Это даёт
> чистые компоненты страниц, в которых нет логики работы с API.

## Слайд 11. Безопасность и валидация

**Тезисы:**

- Аутентификация через Django session + httponly-cookie.
- CSRF-проверка отключена для API; защита через CORS allowlist.
- Серверная валидация через DRF-сериализаторы.
- На фронте — клиентская валидация HTML5 + отображение ошибок поля.
- `IsAuthenticated` по умолчанию для всех ViewSet'ов;
  `AllowAny` явно для публичных эндпоинтов (форма заявки).

**Текст речи:**

> По безопасности применены стандартные практики. Аутентификация —
> сессионная через httponly-cookie, что защищает токен сессии от
> JavaScript-доступа. CSRF-проверка отключена на уровне API через
> кастомный класс `CsrfExemptSessionAuthentication` — это упрощает
> работу со SPA, а защита обеспечивается через строгий CORS-allowlist:
> только домены 4040 и 4041 могут отправлять запросы с cookie.
>
> Все DRF-сериализаторы имеют свои методы валидации. Например,
> в форме заявки имя должно быть длиннее одного символа, а телефон
> должен содержать минимум 10 цифр. На клиенте — стандартные HTML5
> атрибуты `required`, `type="tel"`, `type="email"` плюс отображение
> ошибок DRF в виде подписи под полем. По умолчанию все ViewSet'ы
> закрыты `IsAuthenticated`; публичные эндпоинты, такие как создание
> заявки с лендинга, открыты явным образом через `AllowAny`.

- Backend на 7 Django-приложений, более 20 API-эндпоинтов.

## Возможные вопросы комиссии и ответы

### Вопрос: «Почему именно Nuxt 4, а не React/Next.js?»

> Я выбрал Nuxt 4 из соображений экосистемы и зрелости фреймворка
> для маленьких команд. Vue имеет более низкий порог входа и
> предсказуемую систему реактивности, а Nuxt из коробки даёт
> SSR, файловый роутинг, авто-импорт компонентов и удобную работу
> с состоянием через Pinia. Для проекта одного разработчика это
> дало возможность сосредоточиться на бизнес-логике, а не на
> конфигурации сборки.

### Вопрос: «Почему сессионная аутентификация, а не JWT?»

> Для административной панели, которой пользуется ограниченный
> круг сотрудников и которая работает в одном домене, сессионная
> аутентификация проще и безопаснее. JWT добавляет сложности:
> необходимо реализовать refresh-токены, продумать стратегию хранения
> на клиенте, защититься от XSS. Для системы такого масштаба JWT —
> overengineering. При необходимости масштабирования (например,
> мобильное приложение) можно добавить JWT-эндпоинты как альтернативу.

### Вопрос: «Как обрабатываются ошибки на бэке?»

> Все ViewSet'ы используют DRF-сериализаторы с валидацией. При
> ошибке возвращается стандартный формат `{field: [errors]}`,
> который на фронте парсится в стор и отображается под соответствующим
> полем формы. Общие ошибки (`detail`) показываются как toast-уведомление.
> На бэке настроен CORS, что исключает запросы с неавторизованных
> доменов.

### Вопрос: «Как масштабировать систему?»

> Архитектура изначально готова к масштабированию: бэкенд может работать
> через любой ASGI-сервер (Uvicorn, Daphne); PostgreSQL поддерживает
> репликацию; фронтенд может быть развёрнут как статика на CDN после
> `nuxt build`. Для боевого деплоя нужны: Docker-конфигурация
> (планируется в следующей итерации), Nginx как reverse-proxy,
> SSL через Let's Encrypt. При высокой нагрузке возможно вынести
> кеширование агрегатов (финансовая сводка) в Redis.

### Вопрос: «Где тесты?»

> На текущем этапе акцент сделан на функциональность. Заложены
> возможности для добавления тестов: Django-приложения структурированы
> единообразно, во всех есть `tests.py`. Для следующего этапа
> запланированы юнит-тесты на сериализаторы и интеграционные тесты
> на критичные API-сценарии — создание заявки, авторизация,
> финансовая сводка.

### Вопрос: «Что было самым сложным?»

> Самым сложным было выстроить единую дизайн-систему так, чтобы
> бренд был узнаваем во всех 12 секциях лендинга и одновременно
> в админ-панели, при этом без хардкода и дублирования. Решение —
> вынести все цвета и шрифты в CSS-переменные через `@theme` Tailwind 4,
> и максимально декомпозировать UI на параметризованные компоненты.
> Второй сложный момент — реализация бесшовной бегущей ленты
> «Собирай. Думай. Создавай.» — там пришлось разобраться с тонкостями
> CSS-анимаций и расчётом ширины контейнера, чтобы цикл выглядел
> непрерывным на любом размере экрана.
